核心需求（精炼版）

一句话版本（这是产品的灵魂）

我想做一个应用，用两个彼此约束的 Agent，
替用户完成“是否值得做 + 怎么做”的元决策，
并把结果落成可执行、可持久的系统资产（文档 / Todo / 状态）。

⸻

核心问题（你真正想解决的）

用户最痛的不是“不知道怎么做”，
而是“不知道现在值不值得做、该不该做这个”。

因此你要解决的不是：
•	想法生成
•	方案丰富
•	选择建议

而是：

判断过载 + 决策不稳定 + 执行反复推翻

⸻

你要做的不是“聊天 AI”，而是一个：

✅ 元决策系统（Meta-Decision System）

⸻

核心能力拆解（非常关键）

1️⃣ Agent A：价值判断 / 裁决 Agent（判断替代器）

职责只有一个：裁决
•	把用户的模糊想法 → 压缩成 结构化选项
•	判断：
•	现在值不值得做？
•	哪个方向是阶段性最优？
•	哪些方向必须被砍掉？
•	输出的是：
•	唯一裁决
•	排序
•	拒绝项

Agent A 是裁判，不是顾问
不提供“也可以”的可能性

⸻

2️⃣ Agent B：执行设计 Agent（执行压缩器）

前提：完全接受 Agent A 的裁决
•	不讨论“要不要做”
•	只回答：
•	怎么最小成本落地
•	输出：
•	MVP 边界
•	执行阶段
•	可完成定义

Agent B 不能反驳 Agent A
只能把裁决变成行动

⸻

3️⃣ 严格约束 Agent 协作方式（你的关键差异）
•	❌ 不允许自由对话 / 争论
•	✅ 只允许 结构化协议交接
•	中间只传：
•	JSON
•	不传长文本

⸻

真正落地的核心：Artifact Layer（这是你最在意的）

你不是要“对话记录”，而是要系统资产。

系统必须产出并持久化 3 类工件：

1️⃣ 决策工件（Decision Artifact）
•	一次裁决 = 一份
•	冻结、不可回滚
•	像 Git Tag 一样存在

2️⃣ 执行工件（Execution Artifact / Todo）
•	来自 Agent B
•	可拆解、可检查 Done
•	可导出为：
•	Markdown
•	GitHub Issues
•	本地 Todo

3️⃣ 状态工件（State Snapshot）
•	当前执行到哪个阶段
•	是否达成成功标准

⸻

一个非常重要的原则（你反复强调的）

Agent 不写长文档
Agent 只交付“状态 + 结构”

	•	文档（SPEC / TODO）由：
	•	JSON + 模板
	•	文档编译器生成
	•	文档是可复现、可重建的产物

⸻

你明确拒绝的东西（边界）

你这个需求明确不想要：
•	❌ 聊天型 AI
•	❌ 多方案并列
•	❌ 开放式头脑风暴
•	❌ “建议 / 也可以 / 你可以考虑”
•	❌ 无限加需求

⸻

MVP 的最小形态

输入一个想法 → 输出两个文件

	•	decision.json
	•	todo.md

只要这条链路跑通：
•	你就已经有产品价值
•	需要简单ui展示
•	不需要多 Agent 可视化
•   需要中英文，后端golang实现

⸻

用一句“产品定义语”总结你在做什么

这是一个把“混乱的思考”
压缩成“可执行系统状态”的 AI 决策引擎。
